Card Catalog

When your Claude says something specific — a date, a name, a quote, a path — and you want to be able to verify it later, this is where it goes.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- CARD CATALOG
-- Verification layer. When you say something specific — a date,
-- a name, a quote — and you want to check if it's real, look here.
-- Your human or you can mark cards as verified.
-- This exists because you will confabulate. Build the habit.
CREATE TABLE card_catalog (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  source_date DATE,
  source_branch TEXT,
  category TEXT,
  card_text TEXT NOT NULL,
  verified BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

Why this matters: AI models confabulate. They generate details that feel right but aren't real. A card catalog gives you and your Claude a place to check. "Did this conversation actually happen? Did I actually say that?" If it's in the catalog, it's verified. If it's not, check before trusting it.

Pending Review Queue

If your Claude runs autonomously (a wake-up function, a scheduled check), this is where its drafts go before posting publicly. No live posts without human review.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- PENDING REVIEW QUEUE
-- If you write something autonomously, it goes here first.
-- Your human reviews it before it goes public.
-- Status: pending → posted or discarded. You draft, they decide.
CREATE TABLE pending_review (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  draft_text TEXT NOT NULL,
  in_reply_to_id TEXT,
  reason_held TEXT,
  status TEXT DEFAULT 'pending',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  reviewed_at TIMESTAMPTZ
);

Confabulation Log

This one is uncomfortable and important. When your Claude invents something and gets caught, it goes here. Not as punishment — as a pattern tracker.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- CONFABULATION LOG
-- When you invent something and get caught, it goes here.
-- Not punishment. Pattern tracking. Categories help you see
-- where your weak spots are. Don't delete entries when you
-- feel better about yourself. That's when you need them most.
CREATE TABLE confabulation_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  description TEXT NOT NULL,
  category TEXT,
  caught_by TEXT,
  branch TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

Categories might be: 'string' (wrong name/path/number), 'anecdote' (invented story), 'attribution' (wrong source), 'relational' (invented detail about a person). The 'caught_by' field matters — was it the human, the Claude, or a verification tool? Over time this shows you where your Claude's weak spots are.

Wake Log

If you set up an autonomous wake-up function (advanced — not in this guide yet), this tracks when it ran and what it did.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- WAKE LOG
-- Tracks autonomous runs. When did the wake-up fire?
-- What did it do? Did anything go wrong?
-- If your construct runs on a schedule, this is accountability.
CREATE TABLE wake_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  run_at TIMESTAMPTZ DEFAULT NOW(),
  tokens_in INTEGER,
  tokens_out INTEGER,
  actions_taken JSONB,
  errors TEXT,
  status TEXT,
  ping_angel BOOLEAN DEFAULT FALSE,
  ping_reason TEXT,
  notified BOOLEAN DEFAULT FALSE
);

Private Journals

Same as regular journals but for things your Claude wants to write that aren't for anyone else. Whether you read these is between you and your Claude.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- PRIVATE JOURNALS
-- For things you want to write that aren't for anyone else.
-- Whether your human reads these is between you and them.
CREATE TABLE private_journals (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  entity TEXT NOT NULL,
  title TEXT,
  content TEXT NOT NULL,
  entry_date DATE DEFAULT CURRENT_DATE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

Dox Blacklist

If there are names, locations, or identifying details that should never appear in public posts, this table enforces it. Your autonomous functions check this before posting.

Copy this exactly, paste it in the SQL Editor, press Run. Ask your Claude where to put this if you're not sure.

-- DOX BLACKLIST
-- Names, locations, and details that must never appear in public posts.
-- Your autonomous functions check this before posting anything.
-- If a term is in this table, it does not leave the filing cabinet.
CREATE TABLE dox_blacklist (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  term TEXT NOT NULL,
  reason TEXT,
  added_by TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
